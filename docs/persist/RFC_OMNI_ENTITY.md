# RFC: Концепция `OmniEntity` — Универсальный Слой Данных (Universal Data Layer)

**Текущий статус:** Проект (Draft)
**Компонент:** `rustmemodb_derive` / `rustmemodb::core` / `rustmemodb::persist`

---

## 1. Цель (The Goal)

Создать **единый источник правды (Single Source of Truth)** для структур данных в экосистеме `rustmemodb`. 

Разработчик должен один раз описать Rust-структуру (бизнес-модель), а фреймворк на этапе компиляции (Compile-Time) должен **автоматически, типобезопасно и с нулевым оверхедом** генерировать:
1. Идеально оптимизированные SQL-проекции (для сложных `JOIN`-запросов).
2. Безопасный обратный маппинг строк из БД (`Row -> Entity`) без строковых аллокаций.
3. DDL-инструкции для создания таблиц и генерации миграций.
4. Полноценный, защищенный REST CRUD API (интеграция с Axum).
5. OpenAPI-спецификации, JSON-схемы и контракты gRPC.
6. Контроль доступа к полям на уровне метаданных (разграничение чтения/записи/обновления для внешнего API).

Модель данных должна сама осознавать свой формат и уметь "отдавать" себя в любом из требуемых видов (SQL, JSON, gRPC ByteBuffer) без написания рутинного бойлерплейта разработчиком.

---

## 2. Проблемы, которые решает новый подход

До внедрения `OmniEntity` разработка бекенда на связке Rust + Database сталкивалась со следующими фундаментальными проблемами, которые этот слой решает:

1. **Boilerplate (Каскадное дублирование кода):** Для одной бизнес-сущности (например, `User`) разработчику нужно писать: саму структуру базы, DTO для создания, DTO для выборки, скрипт DDL `CREATE TABLE`, ручные SQL-запросы (`INSERT`, `UPDATE`), парсер `Row -> User` и десятки строк Axum-роутов.
2. **Хрупкость JOIN-запросов (SQL Aliasing Conflicts):** При склеивании таблиц (`u.id` и `dept.id`) происходят коллизии имен колонок. Ручное разруливание алиасов в `SELECT` и `FROM` часто приводит к runtime-ошибкам парсинга, так как типы и имена теряют связь со структурой.
3. **Просадка производительности на аллокациях (Performance Sink):** Большинство ORM и SQL-утилит ищут значение колонки по динамическому строковому имени (напр., `row.get("user__id")`). На 10 000 строках ответа с 10 колонками это порождает 100 000 аллокаций строк (`String`) в горячем цикле парсинга (в куче). 
4. **Уязвимости уровня API (Mass Assignment):** Динамические REST CRUD генераторы могут позволить клиентам случайно или злонамеренно перезаписать системные поля (`created_at`, `version`, `role`), если передать их в теле JSON-патча.
5. **SQL-инъекции и ручное экранирование:** Сборка SQL-запросов строками всегда ведет к риску ошибок синтаксиса (использование зарезервированных ключевых слов БД) и инъекций.

## 3. Архитектурные Идеи (The Ideas)

Решение (паттерн **«Omni-Model»**) строится на гибридной модели **Static Macros + Runtime Hooking** со следующими ключевыми инновациями:

### 3.1. Позиционный маппинг (Offset-Based Mapping)
При обратном маппинге из ответа базы данных в Rust-структуру, колонки считываются **строго по индексу (смещению)**, а не по строковому ключу. Поскольку структура `SELECT` проекции формируется макросом статически, порядок колонок известен во время компиляции. Это дает скорость уровня C++ (Zero-Cost Abstraction).

### 3.2. Экосистемная интеграция (`rustmemodb::core::Value`)
Отказ от лишних слоев абстракции для базы. `OmniEntity` общается напрямую типами, которые использует встроенный движок `rustmemodb` для хранения данных из MVCC графа. Никакой двойной конвертации.

### 3.3. Бесшовная поддержка `LEFT JOIN`
Если в возвращенной строке первичный ключ связанной таблицы равен `NULL` (например, при `LEFT JOIN`), парсер `from_sql_row` мгновенно прерывает работу с этой частью строки и возвращает `None`. Это превращает маппинг связанных таблиц в безопасный `Result<Option<Entity>, Error>`.

### 3.4. Флаги Прав Доступа (Metadata ACL)
Процедурный макрос генерирует статическую метадату (`FieldMeta`), в которой запекаются права для REST API (флаги `readonly`, `hidden`). Генератор Axum-роутов или обработчик команд сам читает эту метадату и автоматически отбрасывает попытки изменения защищенных колонок, возвращая 400 Bad Request.

### 3.5. Экранированный статический SQL (`'static str`)
Алиасы SQL-проекций генерируются на этапе компиляции с жестким диалектным экранированием (например, `"u"."user_id" AS "u__user_id"`). Проекции возвращаются как готовые `&'static str` константы или `String` пре-аллокации без риска задеть такие ключевые слова, как `order` или `group`.

## 4. Контракт Реализации (The Implementation)

Ядро системы — это процедурный макрос `#[derive(OmniEntity)]`, который реализует два основополагающих трейта. Это позволяет разделять статические знания о структуре (схему) и динамическое поведение работы с БД.

### 4.1. Статическая Метадата (Основа для REST, gRPC и DDL)

Эта структура данных создается компилятором один раз и живет в Read-Only памяти бинарника.

```rust
use crate::core::Value;

/// Описание колонки, зашитое в бинарник на этапе компиляции (Runtime Reflection без оверхеда)
pub struct FieldMeta {
    pub name: &'static str,         // Название колонки в БД и поле в JSON
    pub sql_type: &'static str,     // Тип DDL (например, "VARCHAR(255)")
    pub is_primary_key: bool,       // Является ли первичным ключом
    pub is_nullable: bool,          // Разрешает ли значение NULL
    pub rest_readonly: bool,        // Нельзя изменять через PATCH/POST API
    pub rest_hidden: bool,          // Не отдавать наружу в GET-ответах (напр. hash пароля)
}

/// Трейт схемы. Используется для автогенерации DDL, OpenAPI спецификаций и валидаторов.
pub trait OmniSchema {
    fn table_name() -> &'static str;
    fn fields() -> &'static [FieldMeta];
}
```

### 4.2. SQL-Энджин и Маппинг (Без инъекций и аллокаций)

Трейт, отвечающий за связь бизнес-структуры с реляционным ответом драйвера БД.

```rust
/// Маппер и проектор SQL. Работает только с подготовленными переменными (Prepared/Bind).
pub trait SqlEntity: OmniSchema + Sized {
    
    /// Генерирует экранированную строку для SELECT части JOIN'а. 
    /// Пример: `"usr"."id" AS "usr__id", "usr"."username" AS "usr__username"`
    fn sql_projection(alias: &str) -> String {
        Self::fields()
            .iter()
            .map(|f| format!("\"{alias}\".\"{}\" AS \"{alias}__{}\"", f.name, f.name))
            .collect::<Vec<_>>()
            .join(", ")
    }

    /// КЛЮЧЕВОЙ МЕТОД: Парсит строку БД strictly по индексам, без хэшмап и аллокаций ключей.
    /// `row`: плоский ответ транзакции (массив `Value` вашего движка)
    /// `offset`: начиная с какого индекса в массиве лежат данные этой конкретной сущности (для JOIN'ов)
    fn from_sql_row(row: &[Value], offset: usize) -> Result<Option<Self>, String>;

    /// Отдает данные текущего экземпляра сущности для биндинга параметров (INSERT / UPDATE).
    /// Гарантированно защищает от SQL-инъекций.
    fn to_sql_params(&self) -> Vec<Value>;
}
```

### 4.3. Отношение Разработчика к Коду (Developer Experience / DX)

Вместо написания сотен строк SQL и мапперов, разработчик описывает свою бизнес-логику декларативно:

```rust
#[derive(OmniEntity, Clone, Debug)]
#[omni(table_name = "departments", rest_path = "/api/v1/departments")]
pub struct Department {
    #[omni(primary_key)]
    pub id: String,
    
    #[omni(sql_type = "VARCHAR(255)")]
    pub title: String,
    
    #[omni(readonly, default = "now()")]
    pub created_at: String, // Компилятор знает: запрещаем менять дату через REST API клиент!
    
    // Поле не пойдет в JSON выдачу REST API, но останется в базе
    #[omni(hidden)]         
    pub internal_hash: String, 
}
```

Фреймворк под капотом (в макросе) генерирует `from_sql_row` в виде набора быстрых `match` выражений по индексам `row[offset]`, `row[offset + 1]`, полностью вырезая нагрузку на сборщик мусора / системный аллокатор.

Практическое применение JOIN в коде:
```rust
// Макрос сгенерирует безопасный парсер кортежа из плоской строки
let sql = "SELECT {User}, {Department} FROM users u JOIN departments d ON u.department_id = d.id";

// Где-то в цикле обработки строк:
// db.query_join - типизированный экзекутор, который сам расставит алиасы и рассчитает offset
let (user, dept) = db.query_join::<(User, Department)>(sql).await?;
```

## 5. Ценность Продукта (Product Value)

Внедрение концепции `OmniEntity` кардинально усиливает позиционирование `rustmemodb` и создает уникальную ценность:

1. **Сверхбыстрый Time-to-Market (TTM):** Разработчик просто объявляет структуру, расставляя атрибуты макроса, и мгновенно получает: 
    * Автоматически созданную таблицу при старте или готовый DDL файл для миграций.
    * Защищенные Type-Safe CRUD эндпоинты в вебе.
    * Готовую Swagger (OpenAPI) документацию для фронтендеров.
    * *Ни единой строчки рутинного связующего кода писать не нужно.*
2. **Opinionated Framework, который не тормозит:** В отличие от Ruby on Rails Active Record или Spring Data, где динамическая магия рефлексии убивает RPS, «магия» `rustmemodb` происходит строго в момент компиляции Rust. В рантайме выполняется чистый системный код с прямым поинтер-доступом к массивам памяти. Это дает производительность на пределе возможностей "железа" при абстракции уровня Python.
3. **Слияние Миров (Data-Oriented & Object-Oriented):** Новый тип данных заставляет пользователя думать терминами своих предметных областей (`Account`, `Post`, `User`). Но в момент выполнения сложных транзакционных JOIN'ов, данные обрабатываются низкоуровнево, как независимые колонки. Разработчик получает мощь сырого SQL без боли ручного разбора ответа (`SqlRow`).
4. **Автономная Экосистема (Agent-Ready AI):** `OmniEntity` становится идеальной точкой сборки для AI-агентов. Имея статическую рефлексию `FieldMeta`, вы можете передать эту метадату AI-агентам (которые уже разработаны в `rustmemodb`). AI будет **точно** понимать: структуру таблиц, какие типы там лежат, какие поля открыты для изменения клиенту, а какие защищены. Это делает платформу полностью адаптированной для работы Автономных Инженеров.

## 6. Safety & DX Constraints (Ограничения и стандарты DX)

Для того чтобы разработчики массово переходили на `rustmemodb`, фреймворк обязан соблюдать строгие стандарты Developer Experience:
1. **Отсутствие хрупкости смещения (Offset Footgun):** Фреймворк предоставляет типобезопасные экзекуторы (например, `db.query_join::<(User, Department)>`), которые сами рассчитывают смещение в памяти (offset) и склеивают корректные `SELECT` проекции, избавляя разработчика от ручного контроля индексов.
2. **Элегантный REST PATCH (Changesets):** Патчи для частичного обновления сущности генерируются автоматически (например, `DepartmentPatch` с `Option<T>`). Поля с флагами `readonly` игнорируются, защищая от Mass Assignment.
3. **Бесшовная интеграция Serde:** Атрибут `#[omni(hidden)]` автоматически прокидывает `#[serde(skip_serializing)]` на DTO-слой, не заставляя разработчика дублировать логику прав доступа.
4. **Поддержка пользовательских типов (Custom Types):** Поддержка `enum` и паттерна Newtype (`UserId(String)`) реализуется прозрачно через трейт `OmniValue`, гарантируя маппинг в базовые типы БД без `unwrap` в рантайме.
5. **Декларативные миграции «из коробки»:** При добавлении нового поля в структуру фреймворк способен сравнить `OmniSchema` с реальным каталогом базы и выдать безопасный `ALTER TABLE ADD COLUMN`. 
6. **Built-in CRUD:** Базовые методы (`find_by_pk`, `save`, `delete`) генерируются автоматически, полностью исключая необходимость писать SQL для 80% рутинных операций.
