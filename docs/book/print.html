RustMemoDB Documentation

RustMemoDB

RustMemoDB is an embedded persistence and runtime toolkit for Rust applications that want business-logic-first APIs without starting from a separate database service.

The recommended product surface today is rustmemodb::prelude::dx::*. This book is written around the APIs that are currently exercised by examples and tests in this repository.

What You Can Rely On Today

  • #[domain] for persisted domain models.
  • #[api] and #[expose_rest] for generated REST contracts from model methods.
  • PersistApp::open_auto(...) for opening durable embedded storage.
  • serve_domain!(...) for mounting generated routers.
  • PersistJson<T> for nested aggregate state without local wrapper boilerplate.
  • Built-in audit endpoints, generated OpenAPI, typed views, and list query DSL.
  • AgentSessionRuntime and AgentWorkflowExecutor for AI session memory and replay flows.

Who This Fits

  • Rust teams shipping single-binary services, edge apps, or local-first tools.
  • Product codebases that want the domain model, persistence, and HTTP contract to stay close together.
  • Teams that value fast iteration and fewer infrastructure dependencies during the first product phase.

Who This Does Not Fit

  • Teams whose primary requirement is a shared external SQL service for many unrelated applications.
  • SQL-first analytics and warehouse workloads.
  • Systems where cross-language writes to the same store are the main integration model.

Quickstart

If you want the shortest path to a real RustMemoDB success, start with the high-level DX surface:

#![allow(unused)]
fn main() {
use rustmemodb::prelude::dx::*;
}

The current recommended shape is:

  1. Define a model with #[domain(...)].
  2. Put business methods in an #[api] or #[expose_rest] impl.
  3. Open storage with PersistApp::open_auto(...).
  4. Mount the generated router with serve_domain!(...).

Minimal Domain Model

This shape is compile-verified in tests/persist_dx_api_macros_tests.rs and tests/dx_contract_compile_tests.rs.

#![allow(unused)]
fn main() {
use rustmemodb::prelude::dx::*;
use serde::{Deserialize, Serialize};

#[domain(table = "boards", schema_version = 1)]
#[derive(Clone, Debug, PartialEq, Serialize, Deserialize)]
struct Board {
    name: String,
    active: bool,
}

#[derive(Clone, Debug, PartialEq, Eq, DomainError)]
enum BoardError {
    #[api_error(status = 422, code = "validation_error")]
    Validation(String),
}

impl std::fmt::Display for BoardError {
    fn fmt(&self, f: &mut std::fmt::Formatter<'_>) -> std::fmt::Result {
        match self {
            Self::Validation(message) => write!(f, "{message}"),
        }
    }
}

impl std::error::Error for BoardError {}

#[derive(Clone, Debug, PartialEq, Eq, Serialize, Deserialize, Validate)]
struct RenameInput {
    #[validate(trim, non_empty, len_max = 64)]
    name: String,
}

#[api]
impl Board {
    pub fn new(name: String) -> Self {
        Self {
            name: name.trim().to_string(),
            active: false,
        }
    }

    #[command(validate = true)]
    pub fn rename(&mut self, input: RenameInput) -> Result<String, BoardError> {
        self.name = input.name;
        Ok(self.name.clone())
    }

    #[query]
    pub fn current_name(&self) -> String {
        self.name.clone()
    }
}
}

Architecture Overview

RustMemoDB is strongest when you let the domain model be the center of the application.

Recommended Stack

  1. Import rustmemodb::prelude::dx::*.
  2. Declare persisted domain state with #[domain(table = "...", schema_version = ...)].
  3. Put business methods in #[api] or #[expose_rest].
  4. Use PersistJson<T> when the aggregate owns nested JSON-like data.
  5. Open storage with PersistApp::open_auto(...).
  6. Mount generated HTTP with serve_domain!(...).

What The Runtime Generates

  • A persisted model handle for create/get/list/delete and command execution.
  • REST endpoints derived from constructor, command, query, and view methods.
  • List query parsing for pagination, sorting, and field filters.
  • Audit endpoints at /:id/_audits.
  • OpenAPI output at /_openapi.json.
  • Typed PersistView endpoints when you register views.

Why This Matters

The main product win is not a new query language. It is that the domain model, generated handle methods, and generated HTTP contract all come from the same Rust source of truth.

That cuts out the usual drift between:

  • model structs
  • route DTOs
  • hand-written Axum handlers
  • repository layers
  • documentation that describes a different API than the code actually exposes

Separate Surface For AI Memory

AI session memory is a different runtime surface. Use:

  • AgentSessionRuntime
  • AgentWorkflowExecutor
  • AgentTimelineQuery

Escape Hatch

If you need lower-level persistence internals, RustMemoDB also exposes rustmemodb::prelude::advanced::*.

That is an escape hatch, not the default starting point for product code.

Tradeoffs and Limits

This book should help you choose RustMemoDB for the right reasons.

Good Fit

  • Service-local state where app and persistence ship together.
  • Domain-first APIs where business methods should define the HTTP contract.
  • Single-binary deployment targets.
  • Edge, desktop, local-first, and internal product backends.
  • Teams that want audits, generated OpenAPI, and typed views without building a full platform layer first.

Less Good Fit

  • A shared database owned by many applications written in different languages.
  • Analytics-first systems centered on external SQL tooling and warehouse workflows.
  • Environments where a networked SQL server is already a hard requirement.
  • Teams that specifically want raw SQL as the primary public developer surface.

Practical Advice

  • Start with #[domain] and serve_domain!(...) unless you have a concrete reason not to.
  • Use PersistJson<T> for nested aggregate state instead of building manual storage wrappers.
  • Reach for the lower-level runtime or #[persistent] DSL only when you need explicit command/runtime control.
  • Treat examples and tests as the canonical product contract until the book grows broader coverage.

Why These Limits Matter

RustMemoDB becomes much more compelling when it replaces unnecessary layers. It becomes less compelling when you force it to imitate a remote, shared, SQL-first platform that you already have.

Zero-Boilerplate REST APIs

The strongest current RustMemoDB story is domain-first REST generation from plain Rust types and methods.

The recommended surface for this is:

#![allow(unused)]
fn main() {
use rustmemodb::prelude::dx::*;
}

A Real Example Shape

This pattern is exercised by tests/persist_dx_api_macros_tests.rs.

#![allow(unused)]
fn main() {
use rustmemodb::prelude::dx::*;
use serde::{Deserialize, Serialize};

#[domain(table = "macro_boards", schema_version = 1)]
#[derive(Clone, Debug, PartialEq, Serialize, Deserialize)]
struct MacroBoard {
    name: String,
    active: bool,
}

#[derive(Clone, Debug, PartialEq, Eq, DomainError)]
enum MacroBoardError {
    #[api_error(status = 422, code = "validation_error")]
    Validation(String),
}

impl std::fmt::Display for MacroBoardError {
    fn fmt(&self, f: &mut std::fmt::Formatter<'_>) -> std::fmt::Result {
        match self {
            Self::Validation(message) => write!(f, "{message}"),
        }
    }
}

impl std::error::Error for MacroBoardError {}

#[derive(Clone, Debug, PartialEq, Serialize, Deserialize, Validate)]
struct RenamePayload {
    #[validate(trim, non_empty, len_max = 32)]
    name: String,
}

#[api]
impl MacroBoard {
    pub fn new(name: String) -> Self {
        Self {
            name,
            active: false,
        }
    }

    #[command(validate = true)]
    pub fn rename(&mut self, payload: RenamePayload) -> Result<String, MacroBoardError> {
        self.name = payload.name;
        Ok(self.name.clone())
    }

    pub fn activate(&mut self) -> bool {
        self.active = true;
        self.active
    }

    #[query]
    pub fn is_active(&self) -> bool {
        self.active
    }
}
}

The Invisible Database

RustMemoDB is most compelling when persistence feels like part of the application runtime instead of a separate networked system you have to design around on day one.

What “Invisible” Means Here

The recommended path opens persistence inside the same Rust process as the app:

#![allow(unused)]
fn main() {
let app = PersistApp::open_auto(data_dir).await?;
let boards_router = rustmemodb::serve_domain!(app, Board, "boards")?;
}

That changes the developer experience in concrete ways:

  • no separate database server for the default path
  • one deployable binary plus a data directory
  • domain logic, persistence, audits, and generated HTTP stay in one runtime

What Is Verified In This Repository

  • durable reopen and recovery through PersistApp::open_auto(...)
  • generated REST routers over persisted domain models
  • built-in audit endpoints
  • typed views and query DSL on top of persisted state

You can see those behaviors in:

  • examples/managed/persistence_demo.rs
  • examples/agile_board
  • examples/ledger_core
  • tests/persist_app_tests.rs

Why This Is Useful

For many product backends, the hardest part is not raw query power. It is operational drag:

  • provisioning a separate database early
  • keeping route DTOs in sync with the domain model
  • maintaining extra persistence plumbing in app code

RustMemoDB tries to remove that drag for the embedded, service-local case.

What This Page Intentionally Does Not Claim

  • no benchmark table
  • no “always faster than Postgres” statement
  • no claim that a networked SQL server stops being useful

When Postgres Still Wins

  • a shared database for many services
  • SQL-centric analytics and reporting ecosystems
  • cross-language teams writing to the same source of truth
  • workloads where external database operations and tooling are already the product standard

The point of the invisible database story is not “replace every database.” It is “for the right class of Rust application, persistence can feel local again.”

Native AI Memory

RustMemoDB already has a real AI-memory runtime, but the honest current story is session memory, workflow execution, and replay.

This page is intentionally anchored to the APIs exercised in tests/ai_memory_phase1_usage_tests.rs.

Current Runtime Surface

#![allow(unused)]
fn main() {
use rustmemodb::prelude::dx::{
    AgentSessionRuntime,
    AgentSessionRuntimeConfig,
    AgentTimelineQuery,
    AgentWorkflowExecutor,
    AgentWorkflowStep,
};
use serde_json::json;
use std::sync::Arc;

let mut runtime = AgentSessionRuntime::open(
    AgentSessionRuntimeConfig::new(&root),
).await?;

runtime.register_session_command(
    "set_status",
    Arc::new(|state, payload, _ctx| {
        let status = payload
            .get("status")
            .and_then(|value| value.as_str())
            .unwrap_or("unknown");
        let fields = state.fields_object_mut()?;
        fields.insert("status".to_string(), json!(status));
        state.metadata.version = state.metadata.version.saturating_add(1);
        Ok(Vec::new())
    }),
);

let session_id = runtime
    .create_session(None, json!({ "status": "new" }))
    .await?;

let executor = AgentWorkflowExecutor::new(runtime.memory_mut(), session_id.as_str());

let (correlation_id, results) = executor
    .run_with_generated_correlation(vec![
        AgentWorkflowStep::new("set_status", json!({ "status": "planning" })),
        AgentWorkflowStep::new("set_status", json!({ "status": "running" })),
    ])
    .await?;

let replay = runtime
    .replay_envelopes_for_session(
        session_id.as_str(),
        &AgentTimelineQuery::new()
            .with_command_name("set_status")
            .with_correlation_id(correlation_id),
    )
    .await?;
}

What Exists Today

  • Durable session creation.
  • Named commands over session state.
  • Workflow execution with grouped correlation ids.
  • Timeline filtering and replay for a single session.
  • A repo-level usage test that exercises the full flow.

Why This Matters

If you are building an agent runtime, you usually need more than “store some chat turns.” You need:

  • current session state
  • command history
  • replayable timelines
  • a place to correlate a workflow run across multiple steps

That is the current strength of this AI memory surface.

What This Page Does Not Claim

  • built-in vector search
  • automatic semantic chunking
  • a general-purpose RAG stack

Those can be layered on top, but they are not the verified core API today.

Zero-Cost Abstractions

The most believable RustMemoDB abstraction win today is not “magic SQL with no overhead.” It is that your domain model remains the source of truth while the framework generates the repetitive edges around it.

Compile-Verified DX Surface

tests/dx_contract_compile_tests.rs exercises this shape directly:

#![allow(unused)]
fn main() {
let _router = rustmemodb::serve_domain!(
    app,
    CompileContractBoard,
    "compile_contract_boards",
)?;

let _ = handle
    .rename(
        &id,
        RenameInput {
            name: "Next".to_string(),
        },
    )
    .await;

let _ = handle.activate(&id).await;
let _ = handle.view::<CompileContractBoardSummary>();

let _ = handle
    .query()
    .where_eq("active", true)
    .sort_desc("name")
    .per_page(10)
    .fetch()
    .await;
}

What You Do Not Maintain

  • A separate route layer for every command.
  • Hand-written request parsing for every list endpoint.
  • Extra repository plumbing for basic persisted-domain operations.
  • Parallel view projection code for common read models.
  • A second “API model” that drifts away from the business model.

Nested State Stays In Rust

Examples such as examples/agile_board/src/model.rs keep nested aggregate data inside the model with PersistJson<Vec<...>>.

That means the code you read is still ordinary Rust domain logic. You are not forced into a second storage-specific object graph just to persist owned nested data.

What “Zero-Cost” Means Here

In this book, “zero-cost” should be interpreted as a product-DX claim:

  • fewer layers
  • fewer duplicate schemas
  • fewer manual routes
  • fewer contract mismatches

It should not be read as:

  • a promise of zero allocations
  • a claim that every operation is literally a pointer read
  • a statement that raw SQL is the main public API

The real, verified advantage is that the abstraction removes boilerplate without requiring a second application architecture.

Eternal Runtime and Command DSL

The #[domain] path is the default product path for new apps. RustMemoDB also has a lower-level command/runtime surface for teams that want explicit generated command enums and runtime registration.

This page is anchored to tests/persistent_dsl_tests.rs.

A Real #[persistent] Example

#![allow(unused)]
fn main() {
#[rustmemodb::persistent(schema_version = 2, table = "wallet_dsl")]
#[derive(Clone, Debug, Serialize, Deserialize, PartialEq)]
pub struct WalletModel {
    #[sql(index)]
    pub owner: String,
    #[sql]
    pub balance: i64,
}

#[rustmemodb::persistent_impl]
impl WalletModel {
    #[rustmemodb::command]
    pub fn deposit(&mut self, amount: i64) -> rustmemodb::Result<i64> {
        if amount <= 0 {
            return Err(rustmemodb::DbError::ExecutionError(
                "amount must be positive".to_string(),
            ));
        }
        self.balance += amount;
        Ok(self.balance)
    }

    #[rustmemodb::command(name = "rename_owner")]
    pub fn rename_owner(&mut self, owner: String) {
        self.owner = owner;
    }
}
}

What Is Verified Today

  • runtime envelopes from command values
  • persisted command application
  • runtime command registration
  • projection lookup helpers

The tests also verify schema-version-aware runtime behavior.

The One-Binary Backend

Many teams do not need a large backend architecture to start building a serious product. They need one binary, one data directory, one deploy story, and one codebase where the model and API do not drift apart in the first two weeks.

That is one of RustMemoDB’s strongest product stories.

The Pitch

With the high-level DX path, the application shape is small:

  1. define the model with #[domain(...)]
  2. expose business methods with #[api] or #[expose_rest]
  3. open storage with PersistApp::open_auto(...)
  4. mount the generated router with serve_domain!(...)

The real strategic value is not “save a few files.” It is that product momentum stays attached to business logic instead of to infrastructure assembly.

Schema-First Speedrun

Sometimes the best way to win is not to start with a full domain model. Sometimes you need a working API this afternoon so the team can unblock a UI, a data import, or an internal workflow.

RustMemoDB has a real story for that too.

PersistView Read Models

One of the easiest ways to make RustMemoDB feel like a product platform instead of a persistence library is to show typed views.

Views let you keep product-facing read models close to the domain model and expose them as stable endpoints without writing separate projection handlers by hand.

Two Real Shapes

tests/persist_view_mvp_tests.rs exercises both of the current view styles:

  1. a custom computed view
  2. a metric-derived view

Custom Computed View

#![allow(unused)]
fn main() {
#[derive(Clone, Debug, PartialEq, Eq, Serialize, Deserialize, PersistView)]
#[persist_view(model = ViewBoard, name = "status", compute = compute_status_view)]
struct ViewBoardStatus {
    status: String,
}

fn compute_status_view(model: &ViewBoard) -> ViewBoardStatus {
    ViewBoardStatus {
        status: if model.active {
            "active".to_string()
        } else {
            "inactive".to_string()
        },
    }
}
}

Metric-Derived View

#![allow(unused)]
fn main() {
#[derive(Clone, Debug, PartialEq, Eq, Serialize, Deserialize, PersistView)]
#[persist_view(model = MetricsWorkspace, name = "kpis")]
struct MetricsWorkspaceKpis {
    #[view_metric(kind = "copy", source = "name")]
    workspace_name: String,
    #[view_metric(kind = "count", source = "channels")]
    channels_total: i64,
    #[view_metric(kind = "count", source = "campaigns")]
    campaigns_total: i64,
    #[view_metric(kind = "sum", source = "campaigns", field = "spent_minor")]

}
}
