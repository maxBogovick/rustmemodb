A 7-Day Pilot Plan

If a team is curious but skeptical, the best next step is not a months-long migration. It is a small pilot with a real success condition.

This chapter gives you a practical one-week adoption plan.

Day 1: Pick The Right Wedge

Choose one of these starting points:

  • examples/no_db_api if speed matters most
  • examples/agile_board if you want a clean domain-first REST example
  • examples/pulse_studio if you want a product-grade demo with views and idempotency

The rule is simple: pick the example that already looks closest to the product you want to build.

Day 2: Recreate One Core Aggregate

Model one real thing from your product:

  • a workspace
  • a board
  • a ledger
  • a campaign
  • a runbook or mission

Do not model everything. Model the unit that most clearly exposes the value of keeping domain logic and API close together.

Day 3: Add One High-Value Command

Choose a command that matters:

  • launch campaign
  • transfer funds
  • move task
  • hand off run

This is the moment where RustMemoDB should start feeling better than boilerplate-heavy handler code.

Day 4: Add One Read Model

Create one view or dashboard-shaped surface.

This is where PersistView can make the pilot feel much more like a real product and much less like a toy API.

Day 5: Prove Reliability

Show:

  • _openapi.json
  • Idempotency-Key
  • _audits
  • one clean validation or conflict response

If the pilot cannot show these, it is not yet persuasive enough.

Day 6: Compare Against Current Practice

Ask the team:

  • did this remove code or just move it?
  • did the domain model become clearer?
  • did the API become easier to explain?
  • did a useful demo appear faster?

Those are the right pilot questions.

Day 7: Decide

At the end of the week, make one of three decisions:

  1. continue with RustMemoDB for this slice
  2. keep it for prototype/internal-tool/local-first scenarios only
  3. reject it because your real constraints are outside its sweet spot

All three outcomes are valid. The goal is clarity, not forced adoption.

Why This Chapter Promotes The Product Well

A good pilot plan lowers emotional resistance. It tells the reader:

“You do not need a faith-based rewrite. You can test the product on one real slice and learn something useful in a week.”

That is exactly the kind of pragmatic confidence a serious tool should project.
